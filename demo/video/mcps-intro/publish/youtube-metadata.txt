MCP-S: Verifiable Runtime Evidence for MCP Calls

========================================
DESCRIPTION (full)
========================================
The Model Context Protocol (MCP) gave AI agents a standard way to reach real
tools — source control, databases, internal docs, calendars, payment APIs. That
standardization is exactly why it took off. But the standard stops at the
boundary it creates: once an agent tells a tool to do something sensitive, there
is no built-in, portable way to prove who asked, that the request wasn't altered,
or that it isn't a replay of an earlier one. Transport security and OAuth-style
authorization don't give you per-call runtime evidence you can audit later.

MCP-S closes that gap.

MCP-S adds verifiable runtime evidence around every MCP call. Each request can
carry a signature, a nonce, an expiry, authorization context, and a request
hash. Each response is bound back with its own hash. None of it is "trust us"
security — it's evidence a verifier can independently check while the system is
running. Valid requests are accepted; tampered requests, replays, and bad
response bindings are rejected. Provably.

▸ Why it matters
  • Agents are starting to move money, change records, and touch production.
    A single trusted action that can be copied or replayed becomes two bad ones.
  • MCP-S makes tool calls tamper-evident and replay-proof by construction.
  • It's testable: you can assert "tampered request → rejected" in CI, not hope.

▸ It complements authorization — it doesn't replace it
  Authorization decides whether a caller may use a server, tool, or capability.
  Runtime evidence proves what exact MCP call actually happened. MCP-S sits next
  to OAuth/OIDC, not on top of it — and it doesn't try to solve prompt injection
  or police tool catalogs. One job, done well: portable per-call evidence.

▸ Built for enterprise
  • Signing keys can stay in Google Cloud KMS. MCP-S asks KMS to sign — the
    private key is never exported. You get signatures, not raw key material.

▸ Should this be in the MCP standard?
  That's an open question, and it's being worked in the open. The proposal —
  "A Runtime Evidence Profile for MCP tool calls" — argues the natural next layer
  is a small set of standard evidence properties (request/response signatures,
  audience binding, freshness, replay protection, authorization-artifact binding,
  response-to-request binding) that any verifier can check, supporting both
  native-server and sidecar/gateway models. MCP-S is the experimental reference
  implementation showing it's feasible. If you build agent infrastructure, read
  the proposal and weigh in.
