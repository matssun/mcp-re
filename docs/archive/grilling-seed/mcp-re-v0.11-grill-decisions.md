## Settled substance (judge-passed, pending branch sign-off)

- **A.1 — Scope & parity.** All five seed work items (standards ADR, mapping appendix, minimal proof path, vectors, issue list) in v0.11. No draft-02 wire element deprecated until a machine-checked parity gate (18-class positive+negative vector list incl. cross-profile downgrade both directions) is green; deprecation itself = later ADR. New crate `mcp-re-http-profile`; `mcp-re-core` stays transport-agnostic. Handwritten profile-limited RFC 9421/9530 code, golden-vector-gated (S15): RFC worked examples + bidirectional cross-verification against an independent implementation, signature-base bytes compared.
- **A.2 — Taxonomy substance.** Umbrella = evidence model over two carriers (native profile, HTTP standards profile); draft-02 never called "experimental" publicly; seed's canonicalization id spelling is WRONG — code authority `mcp-re-jcs-int53-json-v1` (ids.rs:45); add a seed-regression guard vector. Fail closed on missing/unknown/mismatched profile id/label/tag/version/canon-id; no cross-profile fallback.
- **B.1 — Request signature.** Required covered components exactly @method, @target-uri, content-digest, content-type (+ raw `authorization` and raw `dpop` when present, exactly-once else fail closed). Content-Digest = sha-256 over unencoded content bytes; ANY Content-Encoding on a signed MCP request = fail-closed error. `alg="ed25519"` (RFC 9421 registry; native keeps `Ed25519`); keyid = selection hint only, trust via TrustResolver, no TOFU.
- **B.2 — DPoP composition.** Two mandatory non-substitutable proofs when both configured; keys distinct roles (equality only via explicit deployment profile); MCP-RE signature MUST cover `dpop` when present; full fail-closed matrix.
- **C.1 — Response binding.** Response covers @status, content-digest, content-type + `;req` components @method, @target-uri, content-digest, content-type (Codex tightened: content-type;req required). Request-evidence handle = SHA-256 over the request's RFC 9421 signature base (commits to body digest, method, URI, media type, keyid, sig params, covered set); travels ONLY in the body evidence block, protected via signed content-digest; absence/mismatch fail closed. `server_signer` (TrustResolver identity) stays DISTINCT from `keyid` — seed's `server_signer -> keyid` mapping line is wrong and the mapping appendix must map server_signer → trust-resolver entry.
- **D.1 — Artifact binding substance.** Plural array (required, non-empty, in signed request evidence), one request may bind several independent decision artifacts. Per-type digest rules: DPoP access token via RFC 9449 `ath` (never raw token bytes); mTLS-bound token via RFC 8705 `x5t#S256`; RAR exact-bytes or AS-reference+digest; reference entries keep digest-as-binding + reference-as-cross-audit (v0.6 discipline). Unknown fields/types/duplicates fail closed before dispatch. Bind-not-interpret preserved: core = structure + registry admission; profiles verify semantics.
- **E.1 — Replay.** RFC 9421 created/expires/nonce (+keyid, tag) REQUIRED on every MCP-RE signature; DPoP `jti` never substitutes. Replay key = (resolved_signer_identity, resolved_audience, signature_nonce) — never raw keyid (rotation must not reset replay scope). All cache tiers + fail-closed semantics carry over verbatim. Expected-version policy generalizes to expected-PROFILE policy, no default, startup fail-closed.
- **F.1 — Signed rejection substance.** The HTTP profile is the FIRST signed-rejection implementation and rejection signing is REQUIRED conformance behavior (no unsigned-error fallback for classifiable security rejections). ADR-046's native envelope re-scoped native-only with a concrete named trigger ("first non-HTTP conforming consumer requiring trusted rejection reasons") recorded as an explicit scope split. wire_code authoritative after signature verification; status a signed routing hint; class mismatch = failed verification. Unsigned rejection under require-mcp-re = fail closed + client-local evidence code.
- **G.1 — MRTR over standards handles.** McpMrt two-hash shape kept; handles = SHA-256 over the profile-MANDATED verified signature bases (prior request `mcp-re` base; prior InputRequiredResult response base); body-only carriage in the signed request envelope (NO continuation headers — overrides A1's header sketch); fail-closed on any mismatch, consumed-continuation reuse, Content-Digest-derived handles, or fallback to native JCS hashes.

## Escalation slate — Mats's rulings (E-1…E-12)

**E-1 · One tag scheme (A1/A2/B1/B2/E1/F1 conflict).** Four live values: `mcp-re-http-v1`, `mcp-re-http-standards-v1`, `mcp-re-http-request-v1`, `mcp-re-http-rejection-v1`. Codex's A2 argument: tag identifies the application profile, not direction. → **Griller proposal:** ONE profile-id tag both directions: `tag="mcp-re-http-v1"` ("standards" is redundant in the id; version suffix does the work). Rules out per-direction/per-message-type tags.

**E-2 · Signature labels.** Candidates: `mcp-re`/`mcp-re-response` (A1/A2/G1) vs `mcp-re-req`/`mcp-re-resp` (C1) vs separate `mcp-re-rejection` (F1). → **Griller proposal:** request label `mcp-re`, response label `mcp-re-response`, rejection uses `mcp-re-response` too (a rejection IS a response; @status + wire_code distinguish it — a third label adds vocabulary without a security property).

**E-3 · No new MCP-RE HTTP headers (A1's `mcp-re-profile`/`mcp-re-artifact-binding`/`mcp-re-continuation`, E1's `mcp-re-profile`/`mcp-re-audience`, F1's `mcp-re-rejection` header).** → **Griller proposal:** REJECT all new headers. Profile id already travels signed as the `tag` parameter; artifact bindings, audience tuple, continuation, and the evidence handle travel in the body `_meta` evidence blocks (covered by signed content-digest; survives transport re-expression; audience tuple {scheme,host,port,tenant_id,route_id,realm} doesn't fit @target-uri anyway). Header surface = standard fields only: Content-Type, Content-Digest, Signature-Input, Signature (+ authorization/dpop when present). B1's covered-component list becomes the single normative set; A1/E1 lists corrected at sign-off.

**E-4 · Body evidence-block `_meta` keys.** C1 minted `se.syncom/mcp-re.http.response`. → **Griller proposal:** pair them: request `se.syncom/mcp-re.http.request` (audience, artifact_bindings, continuation), response `se.syncom/mcp-re.http.response` (server_signer, request evidence handle) — same authority pattern as the native keys, `.http.` infix marks the profile.

**E-5 · Request-evidence handle name + digest form (C1; v0.6 split-form convention).** Codex used prefix form `"sha-256:<b64url>"` on a NEW field — the v0.6 ruling made split `digest_alg`/`digest_value` the convention for new security fields (prefix retained for legacy only), and `sha-256:` would be a third spelling next to legacy `sha256:`. → **Griller proposal:** name `request_evidence`; split form `{digest_alg:"sha256", digest_value:"<b64url-no-pad>"}` per the v0.6 convention.

**E-6 · MRTR handle names across profiles (G1).** Same names `previous_request_hash`/`input_required_response_hash` with per-profile derivation (native JCS preimage vs HTTP signature base) — one name, two meanings. → **Griller proposal:** KEEP the names, derivation profile-scoped: the meaning is "hash of the profile's mandated request/response evidence preimage," self-describing via the signed profile id; renaming would fork the implemented, vector-frozen McpMrt shape for zero security gain.

**E-7 · D1 renames of Mats-ratified v0.6 vocabulary.** `authorization_binding`→`artifact_bindings[]`, `authz-system-reference`→`artifact-system-reference`, `authorization_system_id`→`artifact_system_id`, + new `artifact_type` axis with 7 registry tokens. → **Griller proposal:** ACCEPT for the HTTP profile / next wire version only ("artifact" is now the accurate domain term since non-authorization artifacts are bound — S13); draft-02 native stays frozen with the old names; mapping appendix records old→new. The `artifact_type` (what kind of artifact) vs `binding_type` (how bound) axis split mirrors the ratified binding_type-vs-profile separation.

**E-8 · Typed schemas for RAR/DPoP in v0.11 (D1 divergence).** Codex: registry tokens only, no typed Rust schemas anywhere yet. Claude: typed schemas for `rar-authorization-details` + `dpop-proof` in PROFILE crates (not Core) in v0.11. Judge flagged Codex's flat rejection as conservative-flavored (conflates "in Core" with "typed at all") and the deferral has no named trigger. → **Griller proposal (S1):** typed profile-crate schemas + `ath`/`x5t#S256` verification for the OAuth pair (DPoP, RAR) in v0.11 — they are exactly what the SEP audience will test; remaining five stay registry-token-only until a consumer appears (named trigger: first deployment binding that artifact type).

**E-9 · Signed-rejection details (F1).** (a) `error.message = wire_code`: REJECT — message stays human-readable; the frozen token lives only in `error.data.mcp_re_error` (new normative constraint on an existing field, duplicated token, no gain). (b) Status map 400/401/403/409/503: RATIFY as proposed. (c) Client evidence code name `mcp-re.rejection_unsigned`: RATIFY. (d) Tighten conformance wording so "where the server can still sign" cannot read as an unsigned-fallback escape hatch.

**E-10 · Expected-profile policy tokens (E1).** Compound `native-draft-02-and-http-rfc9421-v1` token is awkward. → **Griller proposal:** `accepted_profiles` = explicit SET of profile ids ({`draft-02`}, {`mcp-re-http-v1`}, or both), no default, startup fail-closed — same semantics, no combinatorial tokens.

**E-11 · Corpus home (I1).** `mcp-re-conformance/tests/vectors/http-profile/` (Codex) vs `mcp-re-core/tests/vectors/` sibling (Claude). → **Griller proposal:** Codex's location — core must not grow an HTTP surface even in tests; conformance crate owns the runner and aggregator manifest. Ownership rule: vector corpora live with the crate that owns the verifying profile implementation's conformance runner. HTTP-profile rejections REUSE the frozen `mcp-re.*` wire-code tokens (no parallel namespace — same rule as the client taxonomy).

**E-12 · Public labels & umbrella terms (A2).** "MCP-RE evidence model" / "MCP-RE native profile" / "MCP-RE HTTP standards profile"; public draft-02 phrase "conformance-gated migration bridge and native transport profile"; never "experimental". → **Griller proposal:** ratify as-is; update CONTEXT.md glossary on sign-off (MCP-RE remains transport-agnostic at the evidence-model level; the HTTP profile is a carrier, resolving the glossary tension).

## Proposed CONTEXT.md updates (apply on sign-off)

- **MCP-RE evidence model** — the transport-agnostic security properties (authenticity, integrity, freshness, replay resistance, audience/authorization/response binding, verified security context), independent of carrier.
- **MCP-RE native profile** — the draft-02 `_meta` envelope carrier, any transport; conformance-gated migration bridge; never publicly "experimental".
- **MCP-RE HTTP standards profile** — the RFC 9421 + RFC 9530 (+ DPoP/mTLS/RAR binding) carrier for HTTP transports; SEP-facing target; profile id/tag per E-1.
- **Request evidence handle (`request_evidence`)** — per-profile digest committing to the request evidence preimage (native: JCS preimage; HTTP: RFC 9421 signature base); successor of `request_hash` for the HTTP profile (E-5).
- **Artifact binding** — generalization of authorization binding: bind-not-interpret over external decision artifacts (authorization, PDP/DTR decisions, classifier results, human approvals); vocabulary per E-7.
- _Avoid_: "experimental profile" for draft-02; `mcps-jcs-int53-json-v1` (wrong spelling — code authority is `mcp-re-jcs-int53-json-v1`); `server_signer` ↔ `keyid` conflation; per-direction RFC 9421 tag values.
- **Flagged ambiguity:** seed file `mcp-re-v0.11-seed.md` needs corrections: canonicalization id spelling; "signed rejection" listed among proven properties (designed-only, ADR-046); `server_signer -> keyid` mapping line.
