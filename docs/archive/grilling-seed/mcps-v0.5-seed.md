## 1. Purpose

This document defines the initial discussion seed for MCP-S v0.5.

The goal of v0.5 is not to broaden MCP-S into a general MCP security gateway. The goal is to strengthen MCP-S as a clean, standards-ready message-security profile and explicitly align it with emerging public guidance on MCP security.

MCP-S v0.5 should make the project easier to propose, review, and adopt by clearly answering:

1. What MCP-S directly solves.
2. What MCP-S deliberately does not solve.
3. What conformance evidence proves MCP-S solves its chosen layer.
4. Which adjacent security domains compose with MCP-S but remain outside MCP-S Core.

## 2. Strategic Position

MCP-S should be positioned as:

> A transport-agnostic message security profile for MCP that provides authenticity, integrity, freshness, replay resistance, audience binding, authorization binding, response binding, and verified security context.

MCP-S should not be positioned as:

> A complete MCP security gateway.

The broader MCP security stack includes tool catalog integrity, host approval workflows, sandboxing, output filtering, DLP, vulnerability scanning, and operational monitoring. MCP-S composes with those layers but does not own them.

## 3. Why v0.5 Is Needed

Recent public security guidance for MCP highlights gaps in:

* message authenticity;
* message integrity;
* token/session replay;
* optional or inconsistent authorization;
* missing auditability;
* tool poisoning;
* unsafe dynamic tool discovery;
* weak approval workflows;
* output poisoning;
* insufficient sandboxing;
* poor operational monitoring.

MCP-S already addresses the narrow but critical message-security portion:

* signed request envelopes;
* signed response envelopes;
* signer identity;
* audience binding;
* on-behalf-of binding;
* authorization hash binding;
* issued-at and expires-at timestamps;
* nonce-based replay protection;
* request-hash response binding;
* verified context injection;
* trust resolution;
* optional transport binding.

v0.5 should convert that implicit alignment into explicit specification text, conformance tests, and a public claim matrix.

## 4. v0.5 Design Principle

MCP-S v0.5 must remain method-transparent.

MCP-S Core MUST NOT parse or enforce MCP method semantics such as:

* tools/list;
* tools/call;
* resources/list;
* prompts/list;
* tool descriptor review;
* tool catalog approval;
* tool output inspection;
* prompt injection detection.

MCP-S signs and verifies MCP messages. It does not decide what a tool means, whether a tool is safe, or whether a model should invoke it.

### 5.3 Add a “Composability With Adjacent Profiles” Section

Create a short document:

```text
components/mcps/docs/composability.md
```

This should define how MCP-S composes with, but does not subsume:

1. MCP Tool Catalog Integrity Profile / MTCI.
2. Authorization policy engines.
3. Host approval UX.
4. Runtime sandboxing.
5. Output filtering / DLP.
6. Vulnerability and deployment scanners.
7. SIEM / audit pipelines.

Required language:

> MCP-S can protect messages that carry extension data, but it does not define the semantics of those extensions.

Example:

> A future MCP Tool Catalog Integrity Profile may define signed tool descriptors and descriptor pinning. MCP-S can authenticate and bind the messages carrying those descriptors, but MCP-S Core does not define tool catalog approval or descriptor semantics.

### 5.4 Add v0.5 Claim Matrix

Create or update:

```text
components/mcps/docs/claim-matrix.md
```

Each claim should have:

```text
Claim
Allowed wording
Forbidden wording
Required implementation
Required conformance
Residual risk
```

Example rows:

#### Claim: Message authenticity

Allowed:

> MCP-S verifies that a request or response was signed by a key currently trusted for the declared signer identity.

Forbidden:

> MCP-S proves the signer is a safe agent.

Required conformance:

* valid signed request accepted;
* bad signature rejected;
* unknown signer rejected;
* disabled/revoked signer rejected.

#### Claim: Replay resistance

Allowed:

> MCP-S rejects duplicate fresh requests with the same signer, audience, and nonce within the configured freshness window.

Forbidden:

> MCP-S prevents all duplicate business operations.

Required conformance:

* duplicate nonce rejected;
* expired request rejected;
* replay cache unavailable fails closed;
* declared replay durability tier matches implementation.

#### Claim: Tool safety

Allowed:

> MCP-S does not define tool safety.

Forbidden:

> MCP-S prevents tool poisoning or malicious tool descriptors.

Required conformance:

* no MCP-S Core code path parses tools/list for enforcement.

### 5.5 Add Negative Conformance Tests for Method Transparency

This is important after the MTCI split.

Add conformance tests proving MCP-S Core does not depend on MCP method semantics.

Examples:

```text
- signed tools/list request is treated as an ordinary signed MCP request;
- signed resources/list request is treated as an ordinary signed MCP request;
- signed custom method request is treated as an ordinary signed MCP request;
- MCP-S Core verification does not inspect tool descriptors;
- MCP-S Core verification does not require tool catalog manifests.
```

These tests protect the proposal boundary.

### 5.6 Add Proposal-Ready “MCP-S Scope Statement”

Create:

```text
documents/mcps/proposal-scope.md
```

Suggested text:

> MCP-S defines a transport-agnostic security envelope for MCP messages. It provides cryptographic message authenticity, integrity, freshness, replay resistance, audience binding, authorization binding, and response binding. MCP-S does not define MCP tool catalog governance, host consent UX, tool invocation safety, prompt injection detection, output filtering, or sandboxing. These concerns are part of the broader MCP security stack and may be addressed by separate MCP extensions, deployment profiles, or operational controls that compose with MCP-S.

This statement should appear early in any external proposal.

### 5.7 Add “Operational Guidance” Without Turning It Into Protocol

MCP-S v0.5 may include operational guidance, but it must be clearly non-normative.

Create:

```text
components/mcps/docs/deployment-guidance.md
```

Include:

* use durable replay cache in production;
* declare replay durability tier;
* use bounded trust resolver caching;
* prefer hardware/KMS custody where appropriate;
* use mTLS or equivalent transport binding for production HTTP deployments;
* emit audit events for allow/deny/fail-closed paths;
* pair MCP-S with sandboxing and egress controls for tool execution;
* pair MCP-S with MTCI or another tool-catalog mechanism if dynamic tool discovery is used.

Do not make sandboxing or DLP part of MCP-S Core conformance.

> MCP-S audit events are not a full SIEM schema and do not replace deployment audit policy.

### 5.9 Update ADR Index for v0.5

Add a v0.5 umbrella ADR:

```text
ADR-MCPS-0XX: v0.5 Security Boundary, Threat Coverage, and Proposal Readiness
```

This ADR should state:

* MCP-S Core remains method-transparent.
* Tool catalog integrity is excluded from MCP-S Core.
* MTCI is a separate MCP extension/profile.
* v0.5 adds threat coverage docs, claim matrix, proposal scope, audit vocabulary, and conformance boundary tests.
* v0.5 does not add new signed fields unless a specific gap is found.

## 6. Explicit Non-Goals for v0.5

v0.5 MUST NOT:

* reintroduce signed tool manifests into MCP-S Core;
* parse tools/list in MCP-S Core;
* parse tools/call for authorization decisions in MCP-S Core;
* claim to prevent prompt injection;
* claim to prevent output poisoning;
* claim to sandbox tools;
* claim to provide full RBAC;
* claim to solve token lifecycle management generally;
* claim to replace host approval UX;
* claim to detect vulnerable MCP servers.

## 7. Open Grill Questions

### Scope

1. Is MCP-S Core still method-transparent after all current changes?
2. Are there any remaining tool-catalog types, files, or docs inside the MCP-S repo that should move to MTCI?
3. Does any MCP-S conformance test require knowledge of tools/list?
4. Does any MCP-S claim imply tool safety or tool governance?

### Protocol fields

1. Are current request fields sufficient for NSA-aligned claims?

   * signer
   * on_behalf_of
   * audience
   * authorization_hash
   * nonce
   * issued_at
   * expires_at
   * signature

2. Are current response fields sufficient?

   * request_hash
   * server_signer
   * issued_at
   * signature

3. Do we need any v0.5 field changes, or only documentation/conformance hardening?

Default answer should be: no new fields unless a concrete claim cannot be supported.

### Replay and freshness

1. Do replay cache tiers need to be easier to explain for external reviewers?
2. Should the proposal require a minimum production replay tier?
3. Should single-node replay be allowed in the proposal as a limited deployment profile?

### Authorization

1. Does authorization_hash wording make clear that MCP-S binds an authorization artifact but does not define all policy semantics?
2. Do we need a reference authorization profile in v0.5, or should that remain separate?

### Audit

1. What audit events are normative for MCP-S?
2. Which fields must be emitted for forensic usefulness?
3. What must remain deployment-specific?

### Adjacent profiles

1. How do we describe MTCI without making it look like MCP-S scope creep?
2. Should MTCI be mentioned in the MCP-S proposal only as an example of composability?
3. Should the MTCI repo be cited or omitted from the first MCP-S proposal?

### Proposal readiness

1. What exact claims are we willing to make in the first public MCP-S proposal?
2. What claims are forbidden?
3. What conformance evidence must be green before v0.5 is called proposal-ready?

## 9. Success Criteria

MCP-S v0.5 is successful if an external reviewer can answer:

```text
What does MCP-S secure?
What does MCP-S not secure?
What evidence proves the claims?
How does MCP-S compose with other MCP security work?
Why does MCP-S not compete with tool-catalog or host-safety work?
```

MCP-S v0.5 should make the standard proposal narrower, clearer, and easier to accept.

## 10. Recommended Decision

Proceed with MCP-S v0.5 as a proposal-readiness and boundary-hardening release.

Do not add new MCP method semantics.

Do not add tool catalog enforcement.

Do add the documentation, conformance, claim matrix, and audit vocabulary needed to make MCP-S credible as the message-security layer of the MCP ecosystem.
