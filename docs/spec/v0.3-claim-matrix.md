# MCP-S v0.3 — Security-Claim Matrix

*Release-gate deliverable for [epic #7 / the v0.3 claim-boundary](./v0.3-claim-boundary.md). Maps each declared deployment mode to the exact claim it supports and the claim it forbids. The v0.3 multi-node claim is the AND of the four declared modes — it is only as strong as the weakest declared tier.*

The v0.3 multi-node profile is **tiered, not unconditional**. A deployment declares one tier per axis; the composed claim is read off this matrix. Each cell's wording is the tier's own honest guarantee surfaced by the implementation (`ReplayDurabilityTier::guarantee`, `ResponseSigningIdentityMode::guarantee`, the ADR-021 window, the ADR-023 binding mode) — the proxy cannot surface a claim stronger than its configured tier.

## Axis 1 — Replay-store durability (ADR-MCPS-020)

| Declared tier | Allowed claim | Forbidden claim |
| --- | --- | --- |
| `REDIS_ASYNC` | Replay-safe in steady state across nodes | Replay-safe across a failover / restart-with-state-loss (a window bounded by freshness may reopen) |
| `REDIS_WAIT_QUORUM` | Materially reduced failover replay risk; insufficient acks fail closed | Linearizable / unconditional replay safety |
| `LINEARIZABLE` | Strongest horizontal replay safety, conditional on the store's durable linearizable write contract | Replay safety without that store contract |
| `SINGLE_STORE_FAIL_CLOSED` | Strong **only** under the fail-closed invariant (fleet rejects on store unavailability / possible state loss until the freshness window elapses) | High availability; replay safety while serving during store outage |

Strict-production deployments MUST declare `REDIS_WAIT_QUORUM` or stronger (`meets_strict_production_minimum`).

## Axis 2 — Trust propagation / revocation window `T` (ADR-MCPS-021)

| Declared tier | Allowed claim | Forbidden claim |
| --- | --- | --- |
| `BOUNDED_CACHE` — bounded cache, window `T` (default 60s, ≤ 300s recommended) *(Tier 1, default)* | Revocation enforced fleet-wide within `T`; a key published < `T` ago is not yet trusted | Zero-window / instantaneous revocation; live or push-based revocation |
| `LIVE` — live strong check *(Tier 2)* | Near-zero revocation window: the trust store is consulted on every verification with no positive-trust caching; a revocation is visible on the next request | Zero-window *proven*; high availability (store unavailability fails closed — a hard per-request availability dependency) |
| `PUSH` — push invalidation *(Tier 3)* | Near-zero revocation window with bounded-`T` fallback: a pushed revocation evicts affected entries immediately; on invalidation-channel failure entries still expire within the bounded window `T` | Zero / instantaneous window (the in-process reference channel does not prove reliable ordering/delivery); revocation while the entry is past `T` and the store is down (fails closed) |

ADR-MCPS-021 Tier 2 (`LIVE`) and Tier 3 (`PUSH`) are now **declarable and implemented** (v0.4, issue #70): the in-process reference resolvers (`mcps-proxy` `live_trust::LiveTrustResolver`, `push_trust::PushInvalidationTrustCache`) surface their own honest guarantee via `revocation_tier::RevocationTier::guarantee`, selected with `--revocation-tier {bounded-cache:<T>|live|push:<T>}`. **Zero-window / instantaneous revocation remains a forbidden claim** for every tier: `LIVE` is near-zero at the cost of a hard availability dependency, and `PUSH` is near-zero with a bounded-`T` fallback because the reference channel does not prove reliable ordering/delivery. A networked push backend with proven ordering/delivery would be required — and would be a separate, feature-gated tier — to license a zero-window claim.

## Axis 3 — Signing key custody (ADR-MCPS-022)

| Declared mode | Allowed claim | Forbidden claim |
| --- | --- | --- |
| `per_node_keyset` *(default)* | Per-node Ed25519 keys; one compromised node revokes only that node's key (tight blast radius); server identity is an explicit authorized key set | A single shared identity; smaller blast radius than per-node |
| `shared_remote_signer` | One non-exporting KMS/HSM identity; higher key custody, single pinned identity | Smaller blast radius than per-node (a compromised node credential can sign as the shared identity until revoked) |

Copying one private key across nodes is **forbidden normatively** in every mode.

## Axis 4 — Ingress / transport binding (ADR-MCPS-023)

| Declared mode | Allowed claim | Forbidden claim |
| --- | --- | --- |
| `end_to_end_mtls` | The verified mTLS peer is bound to the request signer end-to-end | — |
| `trusted_ingress_asserted` | Identity asserted by a trusted, authenticated ingress over a trusted hop | End-to-end channel binding; trusting an asserted-identity header from an untrusted hop |

SEP-2243 routing headers (`Mcp-Method` / `Mcp-Name`) are untrusted in **both** modes (ADR-MCPS-025): they never assert identity and never influence a security decision.

## Composition rule

The v0.3 multi-node claim is the **conjunction** of the four declared tiers, bounded by the weakest. Example (the candidate default baseline):

> `REDIS_WAIT_QUORUM` + bounded `T`=60s + `per_node_keyset` + `end_to_end_mtls`
> → *"Multi-node within one trust domain: materially-reduced-failover replay safety (acks fail closed), revocation enforced within 60s, per-node keys with tight blast radius, end-to-end mTLS binding."*

Downgrading any single axis downgrades the composed claim to that axis's row above. No combination licenses any item on the epic's **Not in v0.3** list (multi-tenant isolation between distrusting operators, unconditional async-failover replay safety, zero-window revocation, end-to-end binding under `trusted_ingress_asserted`, smaller-blast-radius-than-per-node for shared KMS, copied private keys, hostile shared-store threat model, cross-operator replay-store isolation).

## Conformance backing

Each tier's behavior is proven by committed conformance tests enumerated in `mcps-conformance/conformance_manifest.json` and re-derived by `//mcps-conformance:drift_guard_test`:

- Replay tiers — `mcps-proxy` `replay_tier` + `shared_replay` / `redis_replay_e2e_test`.
- Trust window — `mcps-proxy` `trust_cache` unit vectors (Tier 1: bounded-`T`, negative-TTL, fail-closed, immediate-evict hook), `live_trust` unit vectors (Tier 2: immediate store-revocation visibility with no `T` wait, store/revocation outage fail-closed), `push_trust` unit vectors (Tier 3: pre-`T` pushed invalidation, channel-failure → bounded-`T` fallback, fail-closed past `T`), and `revocation_tier` unit vectors (per-tier honest `guarantee`, no un-negated zero-window claim) — all under `//mcps-proxy:proxy_unit_test`.
- Key custody — `//mcps-proxy:keyset_admission_test` (per-node accept, unknown/revoked reject, shared-only, mixed-mode disabled, 021+022 composition).
- Ingress — `//mcps-proxy:proxy_transport_test` + the ADR-023 asserted-identity validation.
