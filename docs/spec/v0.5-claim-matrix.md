# MCP-RE v0.5 — Security-Claim Matrix

*Proposal-readiness claim artifact for the v0.5 release ([ADR-MCPS-033](https://github.com/matssun/mcp-re/discussions/381), issue #152). Supersedes [`v0.3-claim-matrix.md`](./v0.3-claim-matrix.md). Two cross-linked sections: **§A** is the reviewer-facing per-capability claim matrix; **§B** is the four-axis deployment-tier matrix. The NSA/threat-coverage matrix is derived from §A — it carries no independent conformance mapping (one evidence spine, [ADR-MCPS-036](https://github.com/matssun/mcp-re/discussions/384)).*

This is a **proposal-readiness** release over the frozen `draft-01` wire envelope. v0.5 adds zero wire-envelope fields; the claim wording is hardened, not the protocol.

## How to read this file

- **§A — Capability-claim matrix.** One row per security property. Every capability is classified as exactly one of two categories — **unconditional** or **deployment-dependent; see §B** — and there is no third category. The Allowed / Forbidden columns are the honest reviewer-facing wording; Required-conformance points at the named green test in [`security_traceability_manifest.json`](../../mcp-re-conformance/security_traceability_manifest.json) ("see the manifest", not a literal count); Residual states what is still not covered.
- **§B — Deployment-tier matrix.** The four-axis composition (replay durability / trust propagation / key custody / ingress binding). A deployment-dependent §A cell links here for its actual strength. The composed claim is the **AND of the declared tiers, bounded by the weakest**.

## §A — Capability-claim matrix

Each row is **either** *unconditional* (Core enforces it the same way in every deployment) **or** *deployment-dependent; see §B* (the strength is a function of declared tiers and is read off §B — the §A cell states no standalone strength). There is no third category. **Tool safety is `none` by design**: MCP-RE Core is method-transparent and makes no claim about whether an invoked tool is safe to run.

| Capability | Category | Allowed claim | Forbidden claim | Required-conformance | Residual |
| --- | --- | --- | --- | --- | --- |
| Message authenticity | **unconditional** | The request/response object was signed by a key in the verifier's authorized key set; a tampered body is rejected over the wire | A valid transport (mTLS) rescues an invalid object signature; identity inferred from an untrusted routing header | Manifest property *"request object signature verified; tampered request body rejected over HTTP"* and *"valid transport identity does NOT rescue an invalid request signature"* | None — Core enforces uniformly |
| Integrity | **unconditional** | Any mutation of the signed payload (body or JSON-RPC `id`) is detected and rejected | Partial-object integrity; integrity of fields excluded from the signing scope | The core vector suite *"tampered body/id … object == http parity"* | Fields outside the canonical signing scope are not covered, by design |
| Signer identity | **unconditional** | The signer is an explicit member of the authorized key set; an unknown/revoked key fails closed | An implicitly trusted signer; first-seen-trust | Manifest property *"server-cert verification: empty server CA fails closed (no implicit trust)"* and the keyset-admission vectors | Key-custody blast radius is **deployment-dependent; see §B Axis 3** |
| Audience binding | **unconditional** — coverage **Direct** | The request is bound to its declared `audience`; a wrong-audience request is rejected | Audience inferred from transport or routing metadata | Manifest property *"core conformance vectors … audience …"* | None — Core enforces `audience` directly (coverage **Direct**) |
| Delegation / authorization binding | **unconditional binding; coverage Partial** | Core **binds** `on_behalf_of` and `authorization_hash` into the signed object; the configured AuthorizationProfile interprets them and decides allow/deny ([ADR-MCPS-013](https://github.com/matssun/mcp-re/discussions/362)) | Core validates artifact contents; Core provides RBAC; Core proves the delegation is legitimate; any `authorization_hash` "mismatch" verdict | Manifest property *"authorization binding: Core binds on_behalf_of and authorization_hash into the signed object"* (Core binds; profile interprets) | Coverage is **Partial**: Core binds but does not interpret — legitimacy of the delegation is a profile/policy decision, not a Core claim |
| Freshness | **unconditional** | A request outside its signed freshness window (expired) is rejected over the wire | Unbounded freshness; freshness without a synchronized clock contract | Manifest property *"freshness: expired request rejected over HTTP"* | None for the single-request check; cross-node replay durability is **deployment-dependent; see §B Axis 1** |
| Replay resistance | **deployment-dependent; see §B Axis 1** | *(see §B Axis 1 — Replay-store durability)* The single-node profile and each multi-node tier surface their own honest guarantee | *(see §B Axis 1)* Unconditional multi-node replay safety; "prevents all duplicate business operations" | Manifest property *"freshness + replay rejected over HTTP on a second submission"*; durable/cross-node strength is read off §B Axis 1 (redis_replay_e2e_test / fleet_replay_e2e_test) | Strength beyond the single-node profile is read off §B Axis 1; nothing here states a standalone strength |
| Revocation / trust propagation | **deployment-dependent; see §B Axis 2** | *(see §B Axis 2 — Trust propagation)* | *(see §B Axis 2)* Zero-window / instantaneous revocation in any tier | Manifest property *"trust-cache / revocation-tier"* (bounded cache admits active, denies unknown; live/push tiers re-derived by drift_guard) | Strength is read off §B Axis 2 |
| Ingress binding | **deployment-dependent; see §B Axis 4** | *(see §B Axis 4 — Ingress / transport binding)* | *(see §B Axis 4)* End-to-end client↔node binding under an asserted-identity tier | Manifest property *"transport-binding EXACT: the verified transport identity must equal the request signer"* | Strength is read off §B Axis 4 |
| Key custody / blast radius | **deployment-dependent; see §B Axis 3** | *(see §B Axis 3 — Signing key custody)* | *(see §B Axis 3)* A shared identity with a smaller blast radius than per-node keys; a copied private key | Manifest property *"keyset-admission"* (per-node accepted, unknown/revoked rejected, shared-remote-signer mode) | Strength is read off §B Axis 3 |
| Response binding | **unconditional** | Each response is signed and bound to its originating request hash, independently verifiable by the client against the server signer's public key | A response accepted against a caller-supplied (un-stored) binding hash; a response bound to the wrong request hash accepted | Manifest property *"response bound to its originating request hash; a response bound to the WRONG request is rejected over HTTP"* and the HostSession stored-hash binding proofs | None — Core enforces uniformly |
| Verified security context | **unconditional** | The proxy is the **sole writer** of `.verified`; any caller-supplied `.verified` is stripped and replaced on every request, even when it was part of the signed payload | A caller-supplied verified-context value is trusted; `.verified` carried across requests of a session | Manifest property *"per-request strip/inject: the proxy is the SOLE writer of .verified"* and *"caller-supplied .verified is stripped and replaced even when it is part of the signed payload"* | None — Core enforces uniformly |
| **Tool safety** | **none, by design** | *(no claim)* — MCP-RE Core is method-transparent and makes no claim about whether an invoked tool is safe to execute | That MCP-RE provides tool safety, sandboxing, or method-semantics enforcement | Method-transparency pair: manifest properties *"method-transparency (ADR-MCPS-030): the Core verdict is independent of the JSON-RPC method"* and *"method-transparency is durable … no … concrete MCP method-name literal"* | Method-aware safety lives **outside** Core (MTCI / policy / host layer), each with its own ADR |

**Classification invariant.** Every row above is in exactly one of `{unconditional, deployment-dependent; see §B}`, except **Tool safety**, whose category is the explicit **`none, by design`** non-goal. No row asserts a standalone strength for a deployment-dependent capability; each instead links to its §B axis.

## §B — Deployment-tier matrix

*Carried forward from the v0.3 multi-node profile ([epic #7 / the v0.3 claim-boundary](./v0.3-claim-boundary.md)) with the v0.4 results folded in (epics #68–#71). This corrects the version-label drift where the `v0.3-…` file carried v0.4 content. Maps each declared deployment mode to the exact claim it supports and the claim it forbids.*

The multi-node profile is **tiered, not unconditional**. A deployment declares one tier per axis; the composed claim is read off this matrix. Each cell's wording is the tier's own honest guarantee surfaced by the implementation (`ReplayDurabilityTier::guarantee`, `RevocationTier::guarantee`, `ResponseSigningIdentityMode::guarantee`, the ADR-023 binding mode) — the proxy cannot surface a claim stronger than its configured tier.

### Replay naming disambiguation

Two distinct things both contain the word "single"; this file never uses "single" loosely for both:

- **Single-node local replay profile** — one proxy instance with a durable local file-backed replay cache; replay-safe **only within that instance**; the sanctioned public entry profile ([ADR-MCPS-017](https://github.com/matssun/mcp-re/discussions/366)). This is the §A *Replay resistance* baseline. It is **not** a multi-node tier.
- **Single shared store fail-closed tier** (`SINGLE_STORE_FAIL_CLOSED`) — a **multi-node** Axis-1 tier using one shared store, valid **only** under the fail-closed invariant (the fleet rejects on store unavailability / possible state loss until the freshness window elapses). This is a §B Axis-1 row, not the single-node profile.

### Axis 1 — Replay-store durability (ADR-MCPS-020)

*Linked from §A → Replay resistance.*

| Declared tier | Allowed claim | Forbidden claim |
| --- | --- | --- |
| `REDIS_ASYNC` | Replay-safe in steady state across nodes | Replay-safe across a failover / restart-with-state-loss (a window bounded by freshness may reopen) |
| `REDIS_WAIT_QUORUM` | Materially reduced failover replay risk; insufficient acks fail closed | Linearizable / unconditional replay safety |
| `LINEARIZABLE` | Strongest horizontal replay safety, conditional on the store's durable linearizable write contract | Replay safety without that store contract |
| `SINGLE_STORE_FAIL_CLOSED` | Strong **only** under the fail-closed invariant (fleet rejects on store unavailability / possible state loss until the freshness window elapses) | High availability; replay safety while serving during store outage |

Strict multi-node production deployments MUST declare `REDIS_WAIT_QUORUM` or stronger (`meets_strict_production_minimum`); the minimum is never relaxed below `REDIS_WAIT_QUORUM`.

`LINEARIZABLE` is **declarable** (epic #68 v0.4 Axis 1, issue #69): a first-class CPStore backend ships — `EtcdAtomicReplayStore` (etcd v3 `lease/grant` + `kv/txn` put-if-absent-under-lease over the synchronous `ureq`/JSON gateway, behind the non-default `cpstore_etcd` cargo feature, ADR-MCPS-018 lean-sync). The claim is **conditional** on etcd's documented durable-linearizable write contract and correct MCP-RE freshness enforcement; it is never over-claimed. Selecting the tier requires `--cpstore-etcd-endpoint` and **fails closed** (a hard config error) if the endpoint is absent or the `cpstore_etcd` feature is not compiled — it is never silently downgraded to Redis or in-memory.

### Axis 2 — Trust propagation / revocation window `T` (ADR-MCPS-021)

*Linked from §A → Revocation / trust propagation.*

| Declared tier | Allowed claim | Forbidden claim |
| --- | --- | --- |
| `BOUNDED_CACHE` — bounded cache, window `T` (default 60s, ≤ 300s recommended) *(Tier 1, default)* | Revocation enforced fleet-wide within `T`; a key published < `T` ago is not yet trusted | Zero-window / instantaneous revocation; live or push-based revocation |
| `LIVE` — live strong check *(Tier 2)* | Near-zero revocation window: the trust store is consulted on every verification with no positive-trust caching; a revocation is visible on the next request | Zero-window *proven*; high availability (store unavailability fails closed — a hard per-request availability dependency) |
| `PUSH` — push invalidation *(Tier 3)* | Near-zero revocation window with bounded-`T` fallback: a pushed revocation evicts affected entries immediately; on invalidation-channel failure entries still expire within the bounded window `T` | Zero / instantaneous window (the in-process reference channel does not prove reliable ordering/delivery); revocation while the entry is past `T` and the store is down (fails closed) |

ADR-MCPS-021 Tier 2 (`LIVE`) and Tier 3 (`PUSH`) are **declarable and implemented** (v0.4, issue #70): the in-process reference resolvers (`mcp-re-proxy` `live_trust::LiveTrustResolver`, `push_trust::PushInvalidationTrustCache`) surface their own honest guarantee via `revocation_tier::RevocationTier::guarantee`, selected with `--revocation-tier {bounded-cache:<T>|live|push:<T>}`. **Zero-window / instantaneous revocation remains a forbidden claim** for every tier: `LIVE` is near-zero at the cost of a hard availability dependency, and `PUSH` is near-zero with a bounded-`T` fallback because the reference channel does not prove reliable ordering/delivery. A networked push backend with proven ordering/delivery would be required — and would be a separate, feature-gated tier — to license a zero-window claim.

### Axis 3 — Signing key custody (ADR-MCPS-022)

*Linked from §A → Key custody / blast radius.*

| Declared mode | Allowed claim | Forbidden claim |
| --- | --- | --- |
| `per_node_keyset` *(default)* | Per-node Ed25519 keys; one compromised node revokes only that node's key (tight blast radius); server identity is an explicit authorized key set | A single shared identity; smaller blast radius than per-node |
| `shared_remote_signer` | One non-exporting KMS/HSM identity; higher key custody, single pinned identity | Smaller blast radius than per-node (a compromised node credential can sign as the shared identity until revoked) |

Copying one private key across nodes is **forbidden normatively** in every mode.

### Axis 4 — Ingress / transport binding (ADR-MCPS-023)

*Linked from §A → Ingress binding.*

| Declared mode | Allowed claim | Forbidden claim |
| --- | --- | --- |
| `end_to_end_mtls` (Tier 1) | The verified mTLS peer is bound to the request signer end-to-end | — |
| `trusted_ingress_asserted` (Tier 2) | Identity asserted by a trusted, authenticated ingress over a trusted hop | End-to-end channel binding; trusting an asserted-identity header from an untrusted hop; **request-binding** (the node cannot verify the assertion was bound to a specific MCP-RE request) |
| `lb_assertion` (Tier 3, ADR-MCPS-023 §"future Tier 3") | **Request-bound ingress assertion**: the node cryptographically verifies (Ed25519 over a length-prefixed canonical preimage) that the trusted LB tied the asserted client identity to **this** request hash, with freshness + a configured LB key-id trust map, failing closed on unknown key / bad signature / request-hash mismatch / stale | **End-to-end client↔node binding** — the LB still terminates the client mTLS and is in the TCB; this is NOT `end_to_end_mtls` |

SEP-2243 routing headers (`Mcp-Method` / `Mcp-Name`) are untrusted in **all** modes (ADR-MCPS-025): they never assert identity and never influence a security decision.

**Tier 3 vs Tier 2 (honesty).** Tier 3 is *strictly stronger* than Tier 2 — it adds cryptographic request-binding the node can verify in hand — but it is *not* Tier 1: the load balancer remains in the trusted computing base because it, not the node, terminates the client's mTLS. Tier 3's guarantee string is `request_bound_ingress_assertion`; it MUST NOT be surfaced as `end_to_end_mtls`. This resolves ADR-MCPS-023's deferred "future Tier 3 boundary" (issue #71). Under `--strict`/`--production` the node refuses to enable `lb-assertion` silently, exactly as it refuses `trusted_ingress_asserted` header ingress.

### Composition rule

The multi-node claim is the **conjunction** of the four declared tiers, bounded by the weakest. Example (the candidate default baseline):

> `REDIS_WAIT_QUORUM` + bounded `T`=60s + `per_node_keyset` + `end_to_end_mtls`
> → *"Multi-node within one trust domain: materially-reduced-failover replay safety (acks fail closed), revocation enforced within 60s, per-node keys with tight blast radius, end-to-end mTLS binding."*

Downgrading any single axis downgrades the composed claim to that axis's row above. No combination licenses any item on the **Not in scope** list (multi-tenant isolation between distrusting operators, unconditional async-failover replay safety, zero-window revocation, end-to-end binding under `trusted_ingress_asserted`, smaller-blast-radius-than-per-node for shared KMS, copied private keys, hostile shared-store threat model, cross-operator replay-store isolation).

## Conformance backing

§A claims map to named green tests via [`security_traceability_manifest.json`](../../mcp-re-conformance/security_traceability_manifest.json) ("see the manifest"). §B tier behavior is proven by committed conformance tests enumerated in [`conformance_manifest.json`](../../mcp-re-conformance/conformance_manifest.json) and re-derived by `//mcp-re-conformance:drift_guard_test`:

- Replay tiers — `mcp-re-proxy` `replay_tier` + `shared_replay` / `redis_replay_e2e_test`.
- Trust window — `mcp-re-proxy` `trust_cache`, `live_trust`, `push_trust`, and `revocation_tier` unit vectors under `//mcp-re-proxy:proxy_unit_test`.
- Key custody — `//mcp-re-proxy:keyset_admission_test` (per-node accept, unknown/revoked reject, shared-only, mixed-mode disabled, 021+022 composition).
- Ingress — `//mcp-re-proxy:proxy_transport_test` + the ADR-023 asserted-identity validation.
